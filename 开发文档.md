## 1. 项目概述

### 1.1 项目目标

构建一个基于强化学习的LLM Agent系统，使Qwen2.5-7B-Instruct模型能够在数学推理任务中：
- 自主进行多步推理（Chain-of-Thought / ReAct）
- 调用外部工具辅助求解（Python执行器、计算器、符号计算）
- 通过GRPO算法优化推理和工具使用策略
- 聚焦**竞赛级数学推理**（MATH + AMC），要求Agent必须综合运用多步推理和工具调用才能解答

### 难度分层设计

GSM8K 题目过简单（小学算术，模型 zero-shot 即可 80%+），用作 RL 训练目标会导致 reward 过早饱和，Agent 学不到深层推理策略。因此采用两层难度体系：

| 层级 | 数据集 | 数量 | 用途 |
|------|--------|------|------|
| **L1: 基础** | GSM8K 子集 | ~1K | 仅用于 SFT 阶段，让模型快速学会工具调用格式 |
| **L2: 核心** | MATH + AMC | ~10K | GRPO 训练主战场，题目需要真正的多步推理和工具综合运用 |

L2 题目特征：代数求解、数论证明、组合计数、不等式推导——这些题必须借助 Python 执行器 / SymPy 才能可靠求解，仅靠模型内化知识正确率有限，因此 RL 训练有明确的优化空间。

### 1.2 技术栈

| 层次 | 选型 | 说明 |
|------|------|------|
| 基座模型 | Qwen2.5-7B-Instruct | BF16 + QLoRA (4-bit) |
| Agent框架 | LangChain + 自研RL包装 | 工具抽象层用LangChain |
| SFT/RFT训练 | TRL (HuggingFace) | SFTTrainer, DPOTrainer |
| RL算法 | GRPO（主力）+ RFT（baseline） | 参考veRL/OpenRLHF实现 |
| 推理引擎 | vLLM | 高效rollout采样 |
| 深度学习框架 | PyTorch 2.x | — |
| 实验管理 | Wandb | 损失、奖励、成功率追踪 |
| 工具执行 | subprocess沙箱 | 简单可控 |
| Python版本 | 3.10+ | — |

### 1.3 硬件配置

| 配置项 | 规格 |

| GPU | RTX 4090 24G × 1（云） |
| 训练精度 | QLoRA 4-bit + BF16 计算 |

┌─────────────────────────────────────────────────────────┐

│     ┌──────────────────────────────────────┐             │

│     └──────────────┬───────────────────────┘             │
│                    ▼                                      │
│  2. 奖励计算                                              │
│     ┌──────────────────────────────────────┐             │
│     │ Reward = outcome_reward              │             │
│     │        + format_reward               │             │
│     │  每条轨迹返回 scalar reward           │             │
│     └──────────────┬───────────────────────┘             │
│                    ▼                                      │

│     ┌──────────────────────────────────────┐             │
│     │ 每组8条轨迹:                          │             │
│     │ A_i = (R_i - mean(R_group))          │             │
│     │       / std(R_group)                 │             │
│     └──────────────┬───────────────────────┘             │
│                    ▼                                      │
│  4. 策略更新（QLoRA参数）                                  │
│     ┌──────────────────────────────────────┐             │
│     │ GRPO loss = -E[π/π_old * A           │             │
│     │            - β * KL(π||π_ref)]       │             │
│     │ 使用AdamW更新LoRA权重                  │             │
│     └──────────────────────────────────────┘             │
│                                                          │
│  每 N 步: 验证集评估 + checkpoint保存                      │
└─────────────────────────────────────────────────────────┘

## 3. 模块详细设计

### 3.1 Agent层

#### 3.1.1 ReAct Agent (`agent/react_agent.py`)

**职责**：实现 Thought → Action → Observation 循环，是Agent的核心执行引擎。

```
ReActAgent
├── 属性
│   ├── model: LLM推理接口
│   ├── tools: List[Tool] 可用工具列表
│   ├── memory: AgentMemory 短期+长期记忆
│   ├── planner: TaskPlanner 任务规划器
│   ├── max_turns: int = 10 最大推理轮次
│   └── system_prompt: str 系统提示词
│
├── 方法
│   ├── run(question: str) → AgentTrajectory
│   │   主入口，执行完整推理并返回轨迹
│   │
│   ├── _think(history: List[Message]) → Thought
│   │   调用LLM生成下一步思考或动作
│   │
│   ├── _act(action: ToolCall) → Observation
│   │   执行工具调用，返回观察结果
│   │
│   ├── _should_stop(thought: Thought, turn: int) → bool
│   │   判断是否结束推理（输出答案或超时）
│   │
│   └── _extract_answer(trajectory: AgentTrajectory) → str
│       从轨迹中提取最终答案
│
└── 输出
    └── AgentTrajectory
        ├── question: str
        ├── steps: List[Step] 每步含 thought, action, observation
        ├── final_answer: str
        └── total_turns: int
```

**关键设计决策**：

- **动作格式**：采用结构化的特殊标记，便于解析：
  ```
  <tool_call>
  {"name": "python_exec", "args": {"code": "..."}}
  </tool_call>
  ```
- **答案格式**：`<answer>最终答案</answer>`
- **停止条件**：
  1. LLM输出包含 `<answer>` 标签
  2. 达到 `max_turns`
  3. 连续3次工具调用失败

#### 3.1.2 Planner (`agent/planner.py`)

**职责**：在推理开始前对问题进行初步分析和步骤规划。

```
TaskPlanner
├── 方法
│   ├── plan(question: str, model) → Plan
│   │   生成初始推理计划
│   │
│   └── revise(plan: Plan, observation: str, model) → Plan
│       根据新观察调整计划（可选）
│
└── 输出
    └── Plan
        ├── original_question: str
        ├── steps: List[str]  规划的子步骤
        └── suggested_tools: List[str]  建议使用的工具
```

**设计说明**：Planner为可选模块。对于GSM8K级别的题目，一步规划就足够；对于MATH竞赛题，可开启多步规划。

#### 3.1.3 Memory (`agent/memory.py`)

**职责**：管理Agent的短期记忆和长期记忆（MD文件持久化）。

```
AgentMemory
├── 短期记忆（上下文窗口内）
│   ├── messages: List[Message]  对话历史
│   ├── tool_results: Dict[str, Any]  工具调用结果缓存
│   └── scratchpad: List[str]  草稿本（临时计算）
│
├── 长期记忆（本地MD文件 + numpy检索）
│   ├── memory_dir: Path  = "data/memory_store/"
│   ├── embeddings: np.ndarray  题目embedding缓存（numpy矩阵）
│   ├── index: Dict[int, Path]  embedding行号 → MD文件路径
│   └── embedder: SentenceTransformer  (all-MiniLM-L6-v2, ~80MB)
│
├── 方法
│   ├── add_message(role, content) → None
│   ├── add_tool_result(tool_name, result) → None
│   ├── get_context() → List[Message]
│   │   返回当前上下文窗口内的消息
│   ├── summarize_old() → str
│   │   对超出窗口的历史进行摘要压缩
│   ├── clear() → None
│   │
│   ├── save_trajectory(trajectory, status, reflection) → Path
│   │   将完整轨迹保存为 MD 文件，自动按来源/子类分目录
│   │   返回文件路径
│   │
│   ├── retrieve_similar(question: str, k=3, status_filter=None) → List[MemoryRecord]
│   │   用 embedding 相似度检索最相关的历史轨迹
│   │   支持过滤：只检索成功案例 / 失败案例
│   │   实现：np.dot(question_emb, embeddings.T) → top-k
│   │
│   ├── build_index() → None
│   │   遍历 memory_store/ 目录，对所有 MD 的 question 计算 embedding
│   │   构建 numpy 矩阵用于快速检索
│   │
│   └── get_few_shot_examples(question, k=2) → List[str]
│       检索最相似的成功案例，格式化为 few-shot prompt 片段
│
└── 设计原则
    ├── 零外部依赖：不需要 Chroma/FAISS/数据库
    ├── 人类可读：所有轨迹以 MD 格式存储，可直接用编辑器打开审查
    ├── 轻量检索：embedding 矩阵存 numpy 内存，cosine similarity 直接 np.dot
    └── 容量上限：万级轨迹无压力（本项目的规模完全够用）
```

**MD 文件格式**：

每条轨迹保存为一个独立的 MD 文件，命名规则：`{dataset}/{subcategory}/{problem_id}_{status}.md`

```markdown
# Problem ID: math_algebra_0012
# Date: 2026-06-13
# Status: success
# Tags: [algebra, quadratic-equation, discriminant]

## Question
求方程 x^2 - 5x + 6 = 0 的所有实数根。

## Ground Truth
x = 2 或 x = 3

## Agent Trajectory
### Step 1
- **Thought**: 这是一个一元二次方程，需要判断判别式，然后求解。
- **Action**: `<tool_call>{"name": "sympy", "args": {"code": "solve(x**2 - 5*x + 6, x)"}}</tool_call>`
- **Observation**: [2, 3]

### Step 2
- **Thought**: 得到两个解 x=2 和 x=3，均为实数根。
- **Action**: (none)
- **Observation**: (none)

## Final Answer
<answer>x = 2 或 x = 3</answer>

## Performance
- Total Turns: 2
- Tool Calls: 1 (1 success, 0 failed)
- Correct: true

## Reflection (失败时填写)
- 错误类型: N/A
- 原因分析: N/A
- 改进建议: N/A
```

**检索流程**：

```
1. 启动时 / 新增轨迹后 → build_index()
   - 遍历 data/memory_store/**/*.md
   - 提取每个文件的 ## Question 块
   - 用 all-MiniLM-L6-v2 做 embedding
   - 存入 np.ndarray (N, 384)

2. 需要检索相似题时 → retrieve_similar(question, k=3)
   - emb = embedder.encode(question) → (384,)
   - scores = np.dot(embeddings, emb) / (norm(embeddings) * norm(emb))
   - top_k_indices = np.argpartition(scores, -k)[-k:]
   - 读取对应 MD 文件内容的 Markdown 文本
   - 返回 MemoryRecord 列表

3. 构造 few-shot → get_few_shot_examples(question, k=2)
   - 调用 retrieve_similar(question, k, status_filter="success")
   - 将成功案例拼接为 <example>...</example> 插入 prompt
```

**MemoryRecord 数据结构**：

```
@dataclass
class MemoryRecord:
    id: str
    question: str
    trajectory_md: str        # 完整 MD 内容
    status: str               # "success" | "failure"
    tags: List[str]
    similarity_score: float
    file_path: Path
```

#### 3.1.4 Action Parser (`agent/action_parser.py`)

**职责**：解析LLM原始输出，提取工具调用或最终答案。

```
ActionParser
├── 方法
│   ├── parse(llm_output: str) → ParsedAction
│   │   从LLM输出中解析动作
│   │
│   ├── is_final_answer(llm_output: str) → bool
│   │   判断是否包含最终答案标记
│   │
│   └── extract_answer(llm_output: str) → str
│       提取最终答案字符串
│
└── 输出
    └── ParsedAction (Enum)
        ├── TOOL_CALL(name, args)
        ├── FINAL_ANSWER(value)
        ├── THOUGHT(text)  纯推理文本
        └── PARSE_ERROR(reason)
```

**解析策略**：
1. 正则匹配 `<tool_call>...</tool_call>` 提取JSON
2. 正则匹配 `<answer>...</answer>` 提取答案
3. 两者都不匹配 → 视为普通推理文本（THOUGHT）
4. JSON解析失败 → PARSE_ERROR，给予-0.1格式惩罚

### 3.3 RL训练层

#### 3.3.1 轨迹定义

```python
@dataclass
class AgentTrajectory:
    question: str                  # 问题文本
    ground_truth: str              # 标准答案
    steps: List[Step]              # ReAct步骤序列
    final_answer: str              # Agent给出的最终答案
    total_turns: int               # 消耗轮次
    is_correct: bool               # 是否正确

@dataclass
class Step:
    turn: int
    thought: str                   # LLM推理文本
    action: Optional[ToolCall]     # 工具调用（如有）
    observation: Optional[str]     # 工具返回结果（如有）
```

#### 3.3.2 Reward 函数 (`rl/reward.py`)

```
RewardFunction
├── 配置
│   ├── outcome_weight: float = 1.0
│   ├── format_weight: float = 0.1
│   └── step_penalty: float = 0.0  每步效率惩罚（可选）
│
├── 方法
│   ├── compute(trajectory: AgentTrajectory) → float
│   │   计算完整轨迹的奖励
│   │
│   ├── _outcome_reward(is_correct: bool) → float
│   │   结果奖励: 正确=1.0, 错误=0.0
│   │
│   ├── _format_reward(trajectory) → float
│   │   格式奖励: 工具调用格式正确 +0.05
│   │            答案格式正确 +0.05
│   │            无解析错误 -0.1 per error
│   │
│   └── _efficiency_penalty(total_turns: int) → float
│       效率惩罚（可选）: -0.01 per step
│
└── 数学答案等价判断 (math_judge.py)
    └── MathJudge
        ├── 数值比较: 相对误差 < 1e-6
        ├── 表达式等价: SymPy simplify(e1 - e2) == 0
        ├── 集合比较: set比较
        └── 分数比较: Fraction归一化
```

**奖励设计细节**：

```
总奖励 = outcome_reward × 1.0 + format_reward × 0.1

outcome_reward:
  - 答案正确: +1.0
  - 答案错误:  0.0

format_reward (范围约 -0.2 ~ +0.1):
  - 每步工具调用JSON格式正确: +0.01
  - 输出包含 <answer> 标签: +0.05
  - 工具调用JSON解析失败: -0.1 / 次
  - 连续无效输出: -0.1 / 次
```

#### 3.3.3 轨迹采样器 (`rl/sampler.py`)

```
TrajectorySampler
├── 属性
│   ├── agent: ReActAgent
│   ├── env: MathEnvironment
│   ├── batch_size: int = 16
│   ├── group_size: int = 8          GRPO每组采样数
│   └── temperature: float = 1.0     采样温度
│
├── 方法
│   ├── sample_batch(problems: List[Problem]) → List[AgentTrajectory]
│   │   对一批问题并行/串行采样轨迹
│   │
│   ├── sample_group(problem: Problem) → List[AgentTrajectory]
│   │   对单个问题采样 group_size 条轨迹
│   │
│   └── _run_single(problem) → AgentTrajectory
│       对单个问题运行一次完整的Agent推理
│
└── 采样策略
    ├── 使用vLLM进行批量推理加速
    ├── temperature=1.0确保探索多样性
    └── 同一题的多条轨迹采用不同随机种子
```

**GRPO采样流程**：

```
对于每个training step:
  1. 随机抽取 batch_size=16 道题
  2. 对每道题，用不同temperature采样 group_size=8 条轨迹
  3. 共获得 16×8=128 条轨迹
  4. 计算每条轨迹的reward
  5. 分组计算advantage
  6. 更新策略
```

#### 3.3.4 GRPO 训练器 (`rl/grpo_trainer.py`)

```
GRPOTrainer
├── 属性
│   ├── policy_model: PeftModel      当前策略（带LoRA）
│   ├── ref_model: PeftModel          参考策略（冻结，用于KL约束）
│   ├── sampler: TrajectorySampler
│   ├── reward_fn: RewardFunction
│   ├── optimizer: AdamW
│   ├── kl_coef: float = 0.04         KL散度系数β
│   ├── clip_eps: float = 0.2          PPO-style clip范围
│   ├── gamma: float = 1.0            折扣因子（数学题不需要折扣）
│   └── max_grad_norm: float = 1.0
│
├── 方法
│   ├── train_step(batch_data) → Dict[str, float]
│   │   执行一步GRPO训练
│   │   返回: {"loss", "reward_mean", "reward_std", "kl_div", "policy_ratio"}
│   │
│   ├── compute_loss(trajectories) → Tensor
│   │   计算GRPO损失
│   │
│   ├── _compute_kl(pi_logprobs, ref_logprobs) → Tensor
│   │   计算KL散度
│   │
│   └── save_checkpoint(path) → None
│
└── 训练循环
    for step in range(max_steps):
        trajectories = sampler.sample_batch(training_data)
        rewards = [reward_fn.compute(t) for t in trajectories]
        loss = compute_loss(trajectories, rewards)
        loss.backward()
        optimizer.step()
        log_to_wandb(...)
```

**GRPO Loss 公式**：

```
对于每个group G（同一道题的8条轨迹）：

  A_i = (R_i - mean(R_group)) / std(R_group)    ← Group相对优势

  L_GRPO = - mean_i [ min( ratio_i * A_i,          ← PPO-style clip
                          clip(ratio_i, 1-ε, 1+ε) * A_i )
                      - β * KL(π_θ(·|s_i) || π_ref(·|s_i)) ]   ← KL惩罚

  ratio_i = exp( log π_θ(a_i|s_i) - log π_old(a_i|s_i) )
```

**关键超参数**：

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| kl_coef (β) | 0.02 ~ 0.05 | 防止策略偏离太远 |
| clip_eps | 0.2 | PPO clip范围 |
| learning_rate | 1e-6 ~ 5e-6 | LoRA用小学习率 |
| group_size | 8 | 平衡效果与效率 |
| batch_size | 16 | 每步题目数 |
| max_grad_norm | 1.0 | 梯度裁剪 |

  # RFT参数
  rft_temperature: 0.8
  rft_filter_ratio: 0.5    # 保留正确率top 50%
  rft_epochs: 2
  rft_rounds: 2

reward:
  outcome_weight: 1.0      # 结果奖励权重
  format_weight: 0.1       # 格式奖励权重
  step_penalty: 0.0        # 步数惩罚（0=不启用）

training:
  seed: 42
  wandb_project: "llm-math-agent"
  output_dir: "./outputs"
  checkpoint_dir: "./checkpoints"

inference:
  backend: "vllm"
  max_tokens: 1024
  temperature: 0.7
  stop_tokens: ["</tool_call>", "</answer>"]

### 6.2 消融实验

| 实验 | 变量 | 探究问题 |
|------|------|----------|
| KL系数 | β ∈ {0.01, 0.02, 0.04, 0.08} | 最佳KL约束强度 |
| Group Size | G ∈ {4, 8, 16} | 组大小对训练的影响 |
| 工具数量 | T ∈ {1, 2, 3} | 工具丰富度的影响 |
| Reward组成 | outcome-only / outcome+format | 格式奖励是否必要 |
| Temperature | τ ∈ {0.8, 1.0, 1.2} | 探索温度的影响 |

### 6.3 错误分析框架

对错误case按以下维度分类：

1. **推理错误**：逻辑链正确，但中间计算或推理步骤出错
2. **工具使用错误**：调用了工具但参数/代码有误
3. **格式错误**：输出格式导致解析失败
4. **答案提取错误**：推理正确但最终答案格式不对
5. **能力边界**：题目本身超出模型能力

## 7. 附录

### 7.1 关键Prompt模板

#### System Prompt

```
你是一个数学解题助手，擅长通过推理和工具使用来解决数学问题。

你可以使用以下工具：

1. calculator - 计算数学表达式
   参数: {"expression": "数学表达式"}
   示例: 计算 123 * 456

2. python_exec - 执行Python代码
   参数: {"code": "Python代码"}
   示例: 计算列表中数字的和

3. sympy - 符号数学计算
   参数: {"code": "SymPy代码"}
   示例: 因式分解 x^2 - 1

请按以下格式进行推理：

1. 分析问题，思考解决方案
2. 如需计算，使用 <tool_call>{"name": "工具名", "args": {...}}</tool_call> 调用工具
3. 根据工具返回的结果继续推理
4. 重复步骤2-3直到得出最终答案
5. 用 <answer>答案</answer> 输出最终结果

注意：
- 最终答案应该放在 <answer></answer> 标签中
- 对于选择题，答案应该只包含选项字母
- 对于数值答案，请使用最简形式（如分数约分、开方化简）
```

#### Few-shot 示例

```
用户: 小明有12个苹果，他给了小红3个，又买了5个，现在有多少个苹果？
助手: 让我逐步推理。

首先计算小明给出苹果后的剩余：12 - 3 = 9

<tool_call>{"name": "calculator", "args": {"expression": "12 - 3"}}</tool_call>
[工具返回]: 9

现在加上购买的苹果：9 + 5 = 14

<tool_call>{"name": "calculator", "args": {"expression": "9 + 5"}}</tool_call>
[工具返回]: 14

小明现在有14个苹果。

<answer>14</answer>
```

### 7.2 常见问题与解决方案

| 问题 | 可能原因 | 解决方案 |
|------|----------|----------|
| 模型不调用工具 | SFT数据不够/温度太低 | 增加SFT数据量，提高温度 |
| 工具调用JSON格式错误 | 模型未学会JSON格式 | 加入更多格式正确的SFT示例 |
| GRPO训练reward不上升 | KL系数太大/学习率太小 | 降低β到0.02，提高lr到5e-6 |
| KL散度爆炸 | 学习率太大/训练步数过多 | 降低lr，早停，增大β |
| vLLM OOM | 显存不足 | 降低max_model_len，减少并发 |
| 答案判定误判 | MathJudge策略不完善 | 添加更多等价性规则 |
| MD记忆检索太慢 | embedding矩阵过大 | 使用 FAISS IVF索引（万级数据可选，常规numpy足够） |
| MD文件过多难以管理 | 未按类别分目录 | 确保按 dataset/category/ 自动分目录 |
| sentence-transformers OOM | 模型加载到GPU | 设置 embedder 在 CPU 上运行（384维速度快） |
